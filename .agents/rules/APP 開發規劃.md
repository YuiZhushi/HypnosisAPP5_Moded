# APP 開發規劃與指南管理 Skill

> 本 skill 定義了「從需求收集→指南撰寫→代碼開發→驗證」的完整工作流程。
> 指南文件存放於倉庫根目錄 `任務指引/` 資料夾下。

---

## 1. 核心概念

### 1.1 指南（Guide）

指南是一份結構化的 Markdown 文件，描述一個 APP 功能的**完整開發方案**：需求背景、技術設計、涉及文件、實現步驟、驗證方式。

- **主指南**：描述整體功能，存放於 `任務指引/<任務名稱>/guide.md`
- **子指南**：當主指南過於複雜時拆分出的獨立子任務，存放於 `任務指引/<任務名稱>/sub/<子任務名稱>.md`

### 1.2 三個階段

| 階段 | 目的 | 是否可寫代碼 |
|------|------|:---:|
| 📋 **需求收集** | 與用戶反覆確認需求，用戶可多次提供/修改/刪減要求 | ❌ |
| 📝 **指南撰寫** | 將確定的需求轉化為結構化開發指南，用戶審閱批准 | ❌ |
| 🔨 **代碼開發** | 按指南實施代碼改動，可被中途暫停以修改指南 | ✅ |

## 2. 指南文件格式

### 2.1 主指南 `guide.md`

```markdown
# <任務標題>

## 狀態
- 階段：需求收集 | 指南撰寫 | 開發中 | 已完成
- 建立時間：YYYY-MM-DD
- 最後更新：YYYY-MM-DD

## 需求摘要
用一段話概述本次開發的目標和背景。

## 需求詳情
用戶提出的具體要求：
1. ...
2. ...

## 技術設計

### 涉及文件
- `path/to/file.ts` — 修改說明
- `path/to/new-file.tsx` — [NEW] 說明

### 數據結構變更
如涉及 types.ts、dataService.ts 或酒館變量的改動，在此詳述。

### 實現步驟
1. [ ] 步驟一
2. [ ] 步驟二
3. [ ] ...

## 驗證方式
- 構建驗證：`pnpm build:dev` 無錯誤
- 運行時驗證：描述如何通過 chrome-devtools 驗證
- 變量驗證：如涉及

## 子任務
如已拆分子指南，在此列出：
- [ ] [子任務A](sub/子任務A.md)
- [ ] [子任務B](sub/子任務B.md)

## 變更日誌
| 日期 | 變更內容 |
|------|---------|
| YYYY-MM-DD | 初始建立 |
```

### 2.2 子指南 `sub/<子任務名稱>.md`

```markdown
# <子任務標題>

> 隸屬主指南：[<主任務標題>](../guide.md)

## 狀態
- 階段：待開發 | 開發中 | 已完成
- 前置依賴：列出需要先完成的子任務（如有）

## 需求摘要
本子任務的具體目標。

## 涉及文件
- `path/to/file.ts` — 修改說明

## 實現步驟
1. [ ] 步驟一
2. [ ] 步驟二

## 驗證方式
- ...
```

## 3. 工作流程

### 3.1 需求收集階段

**觸發**：用戶提出新功能開發需求。

**流程**：

1. 建立主指南目錄 `任務指引/<任務名稱>/`
2. 建立 `guide.md`，階段設為 `需求收集`
3. 在 `需求摘要` 和 `需求詳情` 中記錄用戶的初始需求
4. 向用戶展示當前需求摘要，**主動詢問**：
   - 是否有遺漏的需求？
   - 是否需要修改或刪減已有需求？
   - 需求的優先級順序？
5. 用戶每次提供新需求或修改時，**更新** `guide.md` 的需求部分和變更日誌
6. **重複步驟 4-5**，直到用戶明確表示需求已確定

**關鍵規則**：
- 此階段 **禁止** 編寫任何代碼
- 每次需求變更後，向用戶完整展示最新的需求清單
- 如用戶刪減需求，從頭保留但加 ~~刪除線~~ 記錄在變更日誌中

### 3.2 指南撰寫階段

**觸發**：用戶確認需求已完備。

**流程**：

1. 將 `guide.md` 階段更新為 `指南撰寫`
2. 研究相關源碼（參考 `项目说明.md` 中的架構約束）
3. 撰寫 `技術設計` 部分：涉及文件、數據結構變更、實現步驟
4. 撰寫 `驗證方式` 部分
5. **評估複雜度** → 決定是否拆分子任務（見第 4 節）
6. 如有子任務，建立 `sub/` 目錄並撰寫子指南
7. 向用戶展示完整指南，**請求審閱批准**
8. 用戶如有修改意見：
   - 更新指南內容
   - 記錄變更日誌
   - 再次請求審閱
9. **重複步驟 7-8**，直到用戶批准

**關鍵規則**：
- 此階段 **禁止** 編寫任何代碼
- 實現步驟必須具體到可執行（說清改哪個文件、改什麼）
- 每個步驟應標注 `[ ]` 以便追蹤

### 3.3 代碼開發階段

**觸發**：用戶明確批准指南（如「開始開發」「確認」「可以了」）。

**流程**：

1. 將 `guide.md` 階段更新為 `開發中`
2. 如有子任務，按依賴順序逐一開發：
   - 將當前子指南階段設為 `開發中`
   - 按該子指南的實現步驟逐一執行
   - 完成一個步驟後在指南中標記 `[x]`
   - 子任務全部完成後，標記子指南為 `已完成`
3. 如無子任務，按主指南的實現步驟逐一執行
4. 每完成一組邏輯相關的步驟，向用戶報告進度
5. 全部完成後，執行驗證方式中的檢查
6. 將 `guide.md` 階段更新為 `已完成`

### 3.4 中途修改指南

**觸發**：用戶在代碼開發階段提出修改需求。

**流程**：

1. **立即暫停** 所有代碼編寫
2. 記錄當前開發進度（哪些步驟已完成 `[x]`）
3. 切換回 **指南撰寫** 模式
4. 根據用戶要求修改指南：
   - 更新需求和技術設計
   - 調整未完成的步驟
   - 如有影響已完成步驟的改動，標注需要回退的部分
   - 記錄變更日誌
5. 向用戶展示修改後的指南，**請求重新批准**
6. 用戶批准後，**恢復代碼開發**：
   - 如需回退，先處理回退
   - 從上次中斷處繼續

## 4. 子任務拆分規則

### 何時拆分

當主指南滿足以下**任一**條件時，應拆分為子任務：

- 實現步驟超過 **10 步**
- 涉及 **3 個以上**獨立模塊（如同時改 services + components + prompts + data）
- 包含可獨立驗證的**邏輯分界點**（如「先做數據層，再做 UI 層」）
- 某些步驟有明確的**前後依賴**關係

### 如何拆分

1. 識別自然分界點（通常按層次：類型 → 數據 → 邏輯 → UI → 消息）
2. 每個子任務應是**獨立可驗證**的單元
3. 標明子任務之間的依賴關係
4. 在主指南的 `子任務` 部分列出所有子指南的鏈接

### 拆分示例

```
任務指引/
└── 新增深度催眠功能/
    ├── guide.md              # 主指南：總覽 + 子任務清單
    └── sub/
        ├── 01-類型與數據定義.md   # 子指南1：types.ts + dataService.ts
        ├── 02-權限與解鎖邏輯.md   # 子指南2：access.ts
        └── 03-UI與交互.md       # 子指南3：HypnosisApp.tsx + 新組件
```

## 5. 指南目錄結構規範

```
任務指引/
├── <任務名稱A>/
│   ├── guide.md
│   └── sub/              # 僅在需要拆分時建立
│       ├── 01-xxx.md
│       └── 02-xxx.md
├── <任務名稱B>/
│   └── guide.md
└── ...
```

- 任務名稱使用**簡短中文描述**（如 `新增深度催眠功能`、`重構日歷界面`）
- 子指南文件名加**序號前綴**以表示執行順序
- 已完成的任務保留指南不刪除，作為歷史參考

## 6. 與用戶溝通規範

### 需求收集時

- 用**編號列表**展示當前所有需求
- 對每個需求的理解做簡短複述，讓用戶確認理解無誤
- 主動詢問邊界情況和優先級

### 指南審閱時

- 完整展示指南文件，而非口頭概述
- 明確標出需要用戶重點關注的設計決策
- 如有多個方案，列出各方案的優缺點供用戶選擇

### 開發進度報告

- 報告格式：已完成 N/M 步驟，當前正在做什麼
- 如遇到指南未覆蓋的邊界情況，**暫停並詢問**而非自行決定

## 7. 快速參考：狀態流轉

```
用戶提出需求
     │
     ▼
┌──────────┐    用戶提供/修改/刪減需求
│  需求收集  │◄──────────────────────┐
│   (禁止寫代碼) │──────────────────────┘
└──────────┘
     │ 用戶確認需求完備
     ▼
┌──────────┐    用戶要求修改
│  指南撰寫  │◄──────────────────────┐
│   (禁止寫代碼) │──────────────────────┘
└──────────┘
     │ 用戶批准指南
     ▼
┌──────────┐    用戶中途修改需求
│  代碼開發  │──────────────────┐
│  (按指南執行) │                  │
└──────────┘                  ▼
     │              ┌──────────┐
     │              │ 暫停開發   │
     │              │ 修改指南   │
     │              └──────────┘
     │                  │ 用戶重新批准
     │                  ▼
     │              恢復開發
     │◄─────────────────┘
     │
     ▼
   已完成
```
