# HypnoOS（催眠APP前端）开发说明

> 本说明适用于 `src/催眠APP前端/` 目录树下的开发与修复工作。

---

## 1. 项目概览

HypnoOS 是运行于 SillyTavern **酒馆助手 iframe 环境**中的前端界面项目，以伪手机 OS 风格呈现"催眠"游戏功能。

| 属性 | 值 |
|------|-----|
| 技术栈 | React + TailwindCSS（无 Vue） |
| 打包工具 | 仓库根目录 Webpack（**不使用** 本目录的 Vite 配置） |
| 产物位置 | `dist/催眠APP前端/index.html` |
| 开发命令 | 仓库根目录 `pnpm watch` |
| 构建命令 | 仓库根目录 `pnpm build` 或 `pnpm build:dev` |

## 3. 运行环境假设

代码运行在酒馆助手 iframe 中，以下全局对象**不需要 import，直接可用**：

| 全局对象 | 用途 |
|----------|------|
| `$` | jQuery，用于加载时机与事件绑定 |
| `getVariables` / `updateVariablesWith` / `replaceVariables` | 酒馆助手变量接口 |
| `waitGlobalInitialized` / `eventOn` | 事件与初始化工具 |
| `Mvu` | MVU 变量框架 |
| `createChatMessages` / `triggerSlash` | 发送消息与触发命令 |
| `setChatMessages` / `getChatMessages` | 读写聊天消息 |
| `getCurrentMessageId` / `getIframeName` | 获取上下文信息 |

> ⚠ 纯浏览器直接打开产物无法运行——这是预期行为。

## 4. 数据与持久化契约

### 4.1 用户资源（`UserResources`）→ 聊天变量 `系统`

对应关系由 `dataService.ts` 管理，使用 zod 做默认值与纠错。

| TypeScript 字段 | 酒馆变量路径 |
|-----------------|-------------|
| `mcEnergy` | `系统._MC能量` |
| `mcEnergyMax` | `系统._MC能量上限` |
| `mcPoints` | `系统.当前MC点` |
| `totalConsumedMc` | `系统._累计消耗MC点` |
| `money` | `系统.持有零花钱` |
| `suspicion` | `系统.主角可疑度` |

### 4.2 私有持久化 store（`系统._hypnoos` → `PersistedStore`）

| 字段 | 说明 |
|------|------|
| `debugEnabled` | 调试开关 |
| `sessionEndVirtualMinutes` | 催眠虚拟时间结束点 |
| `sessionEndAtMs` | 催眠现实时间结束点（回退用） |
| `subscription` | 订阅 tier / 到期 / 自动续费 |
| `features` | 各功能开关与 userNote |
| `achievements` / `quests` | 成就领取 / 任务状态 |

**写入规则**：所有对 store 的写入必须通过 `updateStoreWith(...)` 完成，写入后自动调用 `MvuBridge.syncPersistedStore(...)` 做 MVU 同步。

## 5. 架构约束（必须遵守）

### 5.1 组件层不直接操作底层变量

- ✅ 组件只调用 `DataService` 的方法
- ❌ 组件直接调用 `getVariables` / `updateVariablesWith`
- **例外**：`CommonApps.tsx` 内的 `Mvu.events.*` 订阅（仅作为界面刷新触发器）

### 5.2 消息/请求构造统一出口

- 消息构造放在 `prompts/` 目录（如 `buildHypnosisSendMessage`、`buildEditorPipelineParams`）
- ❌ 在组件内手动拼接消息字符串

### 5.3 类型集中管理

- 新增跨模块数据类型→ 放到 `types.ts`
- 在 `DataService` 层做验证/归一化

### 5.4 样式规范

- 优先使用 **TailwindCSS 原子类**
- 避免散落的全局 CSS
- iframe 内禁用 `vh` 单位，用 `width` + `aspect-ratio` 控制高度
- 避免 `min-height`、`overflow: auto` 等撑高容器的属性
- 主体内容不用 `position: absolute`

### 5.5 日志规范

- 统一前缀：`[HypnoOS]`
- 关键节点用 `console.info` 记录
- 可恢复错误用 `console.warn` / `console.error`
- 致命错误用 `throw Error`，顶层函数用 `errorCatched()` 包裹

## 6. 常见改动流程

### 6.1 新增/修改"催眠功能"（Feature）

**涉及文件与步骤**：

1. **`types.ts`** — 如需新字段，扩展 `HypnosisFeature` 接口
2. **`services/dataService.ts`** — 在 `FEATURES` 数组中添加/修改功能定义
   - 设置 `id`、`title`、`description`、`tier`、`costType`、`costValue`、`costCurrency`
3. **`components/HypnosisApp.tsx`** — 功能会按 `tier` 自动分组渲染
   - 如需特殊 UI，在对应位置添加条件渲染
4. **持久化** — 通过 `DataService.updateFeature(...)` 写入 `系统._hypnoos.features`
5. **重置保留** — 如功能在重置后需保留状态，将 id 加入 `PERSISTENT_FEATURE_IDS`

### 6.2 新增一个"桌面应用"

**涉及文件与步骤**：

1. **`types.ts`** — 在 `AppMode` 枚举中添加新值
2. **`components/`** — 创建新组件文件（如 `NewApp.tsx`）
3. **`App.tsx`** — 两处修改：
   - `HomeScreen`：添加 app 图标入口
   - `renderCurrentApp()`：添加路由分发 `case`
4. 如果是**快捷动作**（不需要单独页面），则不需要新 `AppMode`，直接在图标点击事件中执行操作

### 6.3 新增一种消息构造协议

1. **`prompts/`** — 创建纯函数文件构造参数/消息文本
2. **调用处** — 组件只负责触发，不直接手写协议字符串

### 6.4 角色编辑器 / AI 流程变更

1. **`services/characterDataService.ts`** — 角色 YAML 树、行为分支、保存/读取
2. **`services/aiRequestPipelineService.ts`** — API 请求参数、发送、响应处理
3. **`services/characterCompletionApp*Service.ts`** — diff / merge / ai patch
4. **`prompts/characterEditorSend.ts`** — 编辑器请求参数构造

### 6.5 新增成就/任务

1. **`data/questDb.ts`** — 添加任务数据
2. **`services/dataService.ts`** — 如需新的成就判定条件
3. **`components/AchievementApp.tsx`** — 通常自动渲染，特殊 UI 需额外处理

## 8. 修改后验证流程

### 8.1 构建验证

```bash
# 在仓库根目录执行
pnpm build:dev
```

- 确认**无编译错误**
- 确认产物 `dist/催眠APP前端/index.html` 正确生成

### 8.2 类型与接口检查

确保 TypeScript 无类型错误，特别关注：
- `types.ts` / `dataService.ts` / `characterDataService.ts` 是否保持类型一致
- `mvuBridge.ts` 接口调用是否与当前桥接方法一致

### 8.3 运行时验证（通过 chrome-devtools MCP）

如果已经连接了浏览器（酒馆网页），可以通过 chrome-devtools 进行运行时验证：

1. **检查热重载是否可用**
   - 确认酒馆助手的"实时监听"开关已启用
   - 代码变更后界面应自动更新
2. **检查 Console**
   - 确认没有 `[HypnoOS]` 相关的错误日志
   - 确认关键流程有正确的 `console.info` 输出
3. **UI 验证**
   - 截图/快照确认界面渲染正确
   - 验证新增功能的交互是否正常

### 8.4 变量验证

改动涉及变量读写时，使用以下方式验证：
- 通过 chrome-devtools `evaluate_script` 调用 `getVariables({type: 'chat'})` 检查变量值
- 确认写入后的变量格式与 zod schema 定义一致

## 9. 禁止事项

| 禁止 | 原因 |
|------|------|
| 手动修改 `dist/` 目录 | 由打包流程自动生成 |
| 在本目录执行 `npm run dev/build` | 已被禁用，必须用根目录 Webpack |
| 使用 `DOMContentLoaded` | iframe 加载方式不会触发此事件 |
| 使用 `vh` 单位 | 受宿主容器高度影响 |
| 组件直接写变量 | 破坏 DataService 单一数据源原则 |
| 在组件中拼消息字符串 | 应使用 `prompts/` 的纯函数 |
| 使用 `'unload'` 事件 | 应使用 `'pagehide'` |
| 使用 `src=""` 空占位 | 会导致 Webpack 打包错误 |

## 10. 快速参考：关键函数与模式

### 入口挂载模式
```tsx
// index.tsx
$(() => {
  const container = document.getElementById('app');
  if (container) {
    const root = createRoot(container);
    root.render(<App />);
    $(window).on('pagehide', () => root.unmount());
  }
});
```

